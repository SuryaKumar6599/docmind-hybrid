# Strict Implementation Rules
You must adhere to the following architectural constraints and best practices:
1. **Async Worker Pattern:** Next.js functions must NEVER call Ollama, MarkItDown, or LLMLingua directly (Vercel's 10s timeout). Next.js only uploads to Supabase Storage and inserts `pending_processing` rows. The local FastAPI worker polls Supabase, processes the files, and updates the row status. Next.js uses Supabase Realtime to update the UI.
2. **Token-Aware Budget:** Implement `context_manager.py` to allocate tokens (800 system, 1200 schema, 3000 JD, remainder for resume). Only invoke LLMLingua if the text exceeds its specific budget allocation.
3. **Multi-Stage Pipeline & Guardrails:** 
   - Stage 1 (Analytical) and Stage 2 (Creative) MUST use `instructor` wrapping the OpenAI-compatible Ollama client with strict Pydantic schemas (`JobMatchAnalysis`, `TailoredContent`). Never parse raw JSON with regex.
   - Stage 1 must reuse the existing `match_documents()` RPC with `category='portfolio'` to recommend projects.
   - Stage 3 (DOCX/PDF generation) is **pure Python** (`python-docx`). NO LLM calls are allowed to generate document markup.
4. **"Lost in the Middle" Mitigation:** In your prompt templates, ensure the highest-priority extracted facts (missing keywords) go at the very top of the user message, and the candidate's core matching highlights go at the very bottom.
5. **Code Quality:** Use strict type hinting, comprehensive docstrings, modular design, and robust error handling (especially in the FastAPI polling loop to prevent crashes).
