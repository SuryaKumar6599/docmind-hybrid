## 1. Context & Current Architecture
- **Frontend:** Next.js on Vercel. Uploads files to Supabase Storage, inserts `pending_processing` rows, listens via Supabase Realtime.
- **Database:** Supabase Postgres + `pgvector`. Uses `match_documents` RPC for cross-document semantic search.
- **Backend:** Local FastAPI worker on Apple Silicon. Polls Supabase for `pending_processing` rows, processes them using local Ollama, updates the row status to `ready`.
- **Current LLM Stack:** `qwen2.5:7b` for chat, `nomic-embed-text` for embeddings.
- **Current Flaws to Fix:** 
  - The LLM citation parsing uses a hand-rolled `_parse_json` regex fallback instead of strict grammar-constrained generation.
  - The PDF cleaner only does generic whitespace collapsing, missing specific artifact fixes (like de-spacing letters).
  - No vision OCR or advanced structured extraction exists.

## 3. Task C: Agentic Routing & Advanced Extraction
Implement an intelligent routing system so not every document goes through the exact same pipeline.

**Requirements:**
1. **Create `agent_router.py`:**
   - Analyze the file extension and a 500-character text preview.
   - Dynamically route the file to one of three pipelines:
     - `standard_converter`: Clean DOCX/TXT/text PDFs (uses MarkItDown + robust cleaner).
     - `vision_ocr`: Scanned PDFs, images, or files with unreadable text (uses `qwen2.5vl:7b`).
     - `advanced_extractor`: Invoices, contracts, or forms needing strict JSON extraction.
2. **Create `advanced_extractor.py`:**
   - Use `instructor` + `pydantic` to force the LLM (`qwen2.5:7b`) to output a strictly validated JSON schema (e.g., extracting `document_type`, `parties`, `dates`, `amounts`, `obligations`).
   - If the LLM fails to conform, `instructor` should automatically retry up to 3 times.
3. **Update the FastAPI Worker:**
   - Modify the polling loop to call the `AgentRouter` first.
   - Route the file to the appropriate converter based on the decision.
   - Store the routing decision and any advanced extraction JSON in the Supabase row metadata.

## 4. Task D: Local RAG Evaluation Pipeline
Implement a zero-cost, 100% local evaluation framework to prove the RAG system's accuracy.

**Requirements:**
1. **Create `scripts/evaluate_rag.py`:**
   - Do not use paid APIs or heavy cloud frameworks like `ragas`. Use local `qwen2.5:7b` as an "LLM-as-a-Judge".
2. **Define Evaluation Metrics:**
   - **Faithfulness (1-5):** Is the generated answer strictly supported by the retrieved context chunks? (Checks for hallucinations).
   - **Answer Relevancy (1-5):** Does the answer directly address the user's question?
   - **Citation Accuracy (Boolean/Score):** Do the cited `chunk_id`s actually exist in the retrieved context, and do the quotes match?
3. **Test Set Format:**
   - Define a JSONL format for the test set: `{"question": "...", "ground_truth": "...", "context": "..."}`.
4. **Execution Flow:**
   - The script should load the test set, query the Supabase vector store (or use provided context), generate an answer via the `LocalRAG` class, and then pass the (Question, Answer, Context, Ground Truth) to the Judge LLM to get the scores.
   - Output a summary of average Faithfulness and Relevancy scores.
